1. Introduction et contexte industriel

1.1 Problématique métier

Dans toute organisation, la gestion des notes de frais constitue un processus opérationnel critique. Un employé en déplacement avance des dépenses professionnelles — billets de train, repas clients, matériel informatique — et doit en obtenir le remboursement auprès du service financier.

Ce processus, en apparence simple, génère en pratique de nombreux dysfonctionnements lorsqu'il repose sur des moyens informels : e-mails perdus dans les boîtes de réception, pièces jointes égarées, doubles remboursements, absence de traçabilité. Des solutions commerciales comme SAP Concur ou Expensify ont été construites précisément pour répondre à ce besoin. Leur fondation commune : un workflow d'approbation structuré, une gestion des identités et des rôles, un stockage sécurisé des justificatifs, et une interface accessible depuis tout type de terminal.

1.2 Périmètre du projet

Ce projet implémente une version fonctionnelle de cette application en utilisant exclusivement des services AWS gérés, selon une architecture serverless. Deux rôles sont supportés :

  • Employé — soumet des notes de frais, attache des justificatifs, consulte son historique, resoumit après rejet.
  • Responsable financier — consulte la file de validation, approuve ou rejette avec un commentaire obligatoire.

Chaque dépense suit un cycle de vie strict dont les transitions sont contrôlées côté serveur afin de garantir l'intégrité du processus métier, indépendamment des actions du client.

2.2 Choix des services AWS

Service Rôle dans l'application Justification du choix
.NET MAUI Interface Windows. Deux vues distinctes selon le rôle authentifié. Technologie imposée. Base de code unique pour multi-plateforme (Windows, iOS, Android).

Amazon DynamoDB Stockage persistant des notes de frais. Modèle single-table avec PK/SK + 2 GSI. Latence <10 ms, schéma flexible, facturation à la requête. Aucun serveur de base de données.
Amazon S3 Stockage des justificatifs (photos, PDF). Accès uniquement via URLs pré-signées temporaires. Bucket 100 % privé. URLs PUT (15 min upload) et GET (60 min téléchargement). Aucun accès public.
AWS IAM Rôle d'exécution Lambda avec permissions minimales sur DynamoDB et S3. Principe du moindre privilège : chaque Lambda ne peut accéder qu'aux ressources dont elle a besoin.

3. Modèle de données DynamoDB

3.1 Structure de la table principale

Table : expense-tracker-expenses — modèle single-table.

Attribut Type Description
PK (partition key) String USER#<userId> — sub Cognito de l'employé (UUID stable)
SK (sort key) String EXPENSE#<ulid> — ULID pour tri chronologique naturel
ExpenseId String Identifiant unique (ULID)
UserId / UserEmail String Identité de l'employé (sub Cognito + email pour affichage)
Amount Number Montant en euros (décimal)
Category String travel / meals / equipment / other
Description String Libellé libre de la dépense
Status String Draft / Submitted / Approved / Rejected / Resubmitted
ReceiptKey String Clé S3 du justificatif (nullable — présent si upload effectué)
ReviewerComment String Commentaire obligatoire du financier en cas de rejet
CreatedAt / UpdatedAt String Timestamps ISO 8601 (UTC)
Choix ULID vs UUID : L'ULID (Universally Unique Lexicographically Sortable Identifier) encode le timestamp dans ses 10 premiers caractères. Utilisé comme sort key, il garantit un tri chronologique naturel sans index supplémentaire ni colonne CreatedAt dans la SK.

3.2 Index secondaires globaux (GSI)

GSI PK projetée SK projetée Pattern d'accès
StatusIndex Status CreatedAt File de validation Finance : requête sur Status = "Submitted" ou "Resubmitted", triées par date croissante
CategoryIndex Category CreatedAt Reporting analytique par catégorie de dépense sur une période

Le modèle single-table avec PK = USER#id permet de récupérer l'ensemble des dépenses d'un employé par une seule opération Query sans scan coûteux. Le StatusIndex est l'index métier clé : il retourne instantanément la file du service financier, quelle que soit la taille de la table.

3.3 Machine à états

Draft ──[submit]──▶ Submitted ──[approve]──▶ Approved
                               │
                          [reject]
                               │
                               ▼
                    Rejected ──[resubmit]──▶ Resubmitted ──[submit]──▶ Submitted

Toute transition non listée dans la table de transition retourne une erreur HTTP 409 Conflict. La logique est appliquée côté serveur (Lambda), indépendamment du client.

4. Implémentation des éléments clés

4.1 Authentification & JWT Cognito

L'application MAUI utilise le SDK MSAL (Microsoft Authentication Library) pour déclencher le flux OAuth 2.0 / OIDC vers Amazon Cognito. Après authentification, Cognito retourne trois tokens :

  • AccessToken — accès aux ressources Cognito (changement de mot de passe, etc.).
  • IdToken — contient les claims utilisateur (sub, email, cognito:groups). C'est ce token qui est envoyé à API Gateway.
  • RefreshToken — renouvellement silencieux de session.
Erreur fréquente : utiliser l'AccessToken au lieu de l'IdToken provoque une erreur 401 sur API Gateway. Le Cognito Authorizer attend l'IdToken car lui seul contient les claims métier (email, cognito:groups). 
// ExpenseService.cs — envoi du bon token
var authResult = await _publicClientApp
    .AcquireTokenSilent(Scopes, accounts.FirstOrDefault())
    .ExecuteAsync();

// IdToken (et non AccessToken) pour API Gateway Cognito Authorizer
_httpClient.DefaultRequestHeaders.Authorization =
    new AuthenticationHeaderValue("Bearer", authResult.IdToken);

4.2 RBAC — Contrôle d'accès basé sur les rôles

Le JWT émis par Cognito contient le claim cognito:groups. API Gateway injecte les claims décodés dans le contexte de la requête Lambda. La Lambda UpdateExpenseStatus extrait les groupes et vérifie le droit d'action avant tout accès à la base de données :

// Extraction des groupes depuis le contexte injecté par API Gateway
var groups = request.RequestContext.Authorizer.Claims["cognito:groups"]
    .Split(',')
    .Select(g => g.Trim())
    .ToList();

var isFinance = groups.Contains("finance");
var userId    = request.RequestContext.Authorizer.Claims["sub"];

// Actions réservées au service financier (approve, reject)
if (FinanceActions.Contains(action) && !isFinance)
    return Response(403, "Seul le service financier peut approuver ou rejeter.");

// Actions réservées au propriétaire de la dépense (submit, resubmit)
if (OwnerActions.Contains(action) && item["UserId"].S != userId)
    return Response(403, "Vous ne pouvez modifier que vos propres dépenses.");
Action Groupe autorisé Condition supplémentaire
submit employees Propriétaire de la dépense uniquement
resubmit employees Propriétaire de la dépense uniquement
approve finance —
reject finance Champ comment obligatoire dans le body

4.3 Machine à états server-side

Toute tentative de transition invalide est rejetée HTTP 409 Conflict, quel que soit ce que le client demande. Un verrou optimiste DynamoDB (ConditionExpression) prévient les conflits de concurrence entre deux appels simultanés :

// Table de transitions valides — tout ce qui n'y figure pas est refusé
private static readonly Dictionary<(string, string), string> ValidTransitions = new()
{
    { ("Draft",        "submit"),   "Submitted"   },
    { ("Submitted",    "approve"),  "Approved"    },
    { ("Submitted",    "reject"),   "Rejected"    },
    { ("Rejected",     "resubmit"), "Resubmitted" },
    { ("Resubmitted",  "submit"),   "Submitted"   },
};

var currentStatus = item["Status"].S;
if (!ValidTransitions.TryGetValue((currentStatus, action), out var newStatus))
    return Response(409, $"Transition invalide : '{action}' depuis '{currentStatus}'.");

// Verrou optimiste : l'opération échoue si le statut a changé entre la lecture et l'écriture
await _dynamoDb.UpdateItemAsync(new UpdateItemRequest {
    TableName        = TableName,
    Key              = key,
    UpdateExpression = "SET #status = :newStatus, UpdatedAt = :now, ReviewerComment = :comment",
    ConditionExpression = "#status = :currentStatus",   // ← verrou
    ExpressionAttributeValues = new() {
        [":newStatus"]      = new AttributeValue { S = newStatus },
        [":currentStatus"]  = new AttributeValue { S = currentStatus },
        [":now"]            = new AttributeValue { S = DateTime.UtcNow.ToString("o") },
        [":comment"]        = new AttributeValue { S = comment ?? "" },
    },
    ExpressionAttributeNames = new() { ["#status"] = "Status" }
});

4.4 URLs pré-signées S3

Le bucket S3 est entièrement privé (BlockPublicAccess activé). Lambda génère des URLs temporaires via le SDK AWS — le client MAUI peut ainsi uploader ou télécharger des justificatifs sans que le bucket ne soit exposé :

// Upload : URL PUT valable 15 minutes
var uploadUrl = await _s3.GetPreSignedURLAsync(new GetPreSignedUrlRequest {
    BucketName  = _bucketName,
    Key         = $"receipts/{userId}/{expenseId}/{fileName}",
    Verb        = HttpVerb.PUT,
    Expires     = DateTime.UtcNow.AddMinutes(15),
    ContentType = "image/jpeg"
});

// Téléchargement : URL GET valable 60 minutes
var downloadUrl = await _s3.GetPreSignedURLAsync(new GetPreSignedUrlRequest {
    BucketName  = _bucketName,
    Key         = objectKey,
    Verb        = HttpVerb.GET,
    Expires     = DateTime.UtcNow.AddHours(1)
});

La clé S3 est enregistrée dans DynamoDB (ReceiptKey) dès la génération de l'URL d'upload, ce qui permet à l'application de savoir si un justificatif est attaché à la dépense, même avant la fin du transfert.

5. Difficultés rencontrées et solutions adoptées

Problème Cause identifiée Solution
Compilation XAML silencieusement échouée Namespace XML incorrect dans Platforms/Windows/App.xaml (2009/xaml au lieu de 2006/xaml) + propriété WindowsPackageType=None manquante dans le .csproj Comparaison avec un projet template dotnet new maui — correction du namespace et ajout de la propriété
Erreur 401 sur API Gateway L'application envoyait l'AccessToken Cognito au lieu de l'IdToken. Le Cognito Authorizer valide l'IdToken qui contient les claims utilisateur. Correction en une ligne dans ExpenseService.cs : authResult.IdToken à la place de authResult.AccessToken
File Finance incomplète La Lambda GetExpenses ne retournait que les dépenses Submitted, excluant les Resubmitted après correction puis resoumission. Deux requêtes fusionnées sur StatusIndex dans FinanceViewModel, avec déduplification par ExpenseId
ImplicitUsings manquant dans les projets Lambda Types Task<>, Random, List<> non résolus au moment de la compilation car les projets Lambda sont créés avec un template minimal. Ajout de <ImplicitUsings>enable</ImplicitUsings> dans chaque fichier .csproj Lambda
Sérialisation JSON incohérente Noms de champs JSON différents entre les modèles MAUI et les réponses Lambda (expenseId vs ExpenseId) Ajout d'attributs [JsonPropertyName] explicites sur tous les DTOs côté MAUI pour correspondre exactement aux champs DynamoDB

6. Bilan et perspectives

Bilan fonctionnel

L'application déployée sur AWS (eu-west-1) couvre l'intégralité du workflow demandé :

  • Création et gestion de dépenses (Draft → Submitted)
  • Validation ou rejet par le service financier, avec commentaire obligatoire en cas de rejet
  • Resoumission après rejet (Rejected → Resubmitted → Submitted)
  • Upload sécurisé de justificatifs via URLs pré-signées S3 (bucket 100 % privé)
  • Machine à états strictement enforced côté serveur (Lambda) — aucune transition invalide possible
  • RBAC appliqué dès l'entrée dans chaque Lambda, avant tout accès à DynamoDB
  • Interface MAUI avec vues séparées selon le rôle Cognito détecté au login

Améliorations possibles

  • Notifications email via Amazon SES — notifier l'employé lors d'une approbation ou d'un rejet, avec le commentaire du financier inclus dans le mail.
  • Seuil de double validation — exiger deux approbations distinctes pour les montants supérieurs à 500 €, règle configurable via une table DynamoDB de configuration.
  • Dashboard analytique — exploiter le CategoryIndex et Amazon QuickSight pour des rapports de dépenses par catégorie et par période, exportables en PDF.
  • Déploiement mobile iOS/Android — le projet MAUI est structuré multi-plateforme ; seule la configuration de build spécifique à chaque plateforme manque.
  • Infrastructure as Code — migrer le déploiement manuel vers AWS CDK ou Terraform pour une reproductibilité complète de l'environnement.
  • Tests d'intégration — ajouter des tests Lambda avec LocalStack pour couvrir les scénarios de transition d'états et de RBAC sans coût AWS.
