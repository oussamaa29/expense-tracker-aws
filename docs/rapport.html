1. Introduction et contexte industriel

Dans toute organisation, la gestion des notes de frais constitue un processus opérationnel critique. Un employé en déplacement avance des dépenses professionnelles — billets de train, repas clients, matériel informatique — et doit en obtenir le remboursement auprès du service financier. Ce processus, en apparence simple, génère en pratique de nombreux dysfonctionnements lorsqu'il repose sur des moyens informels : e-mails noyés dans les boîtes de réception, pièces jointes perdues, doubles remboursements, absence de traçabilité.

Des solutions commerciales comme SAP Concur ou Expensify ont été construites précisément pour répondre à ce besoin. Leur fondation commune : un workflow d'approbation structuré, une gestion des identités et des rôles, un stockage sécurisé des justificatifs, et une interface accessible depuis tout type de terminal.

Ce projet implémente une version fonctionnelle de cette application en utilisant exclusivement des services AWS gérés, selon une architecture serverless. Deux rôles sont supportés : l'employé qui soumet ses dépenses, et le responsable financier qui les valide ou les rejette avec justification. Chaque dépense suit un cycle de vie dont les transitions sont contrôlées côté serveur afin de garantir l'intégrité du processus métier.

4. Implémentation des éléments clés

4.1 RBAC — Contrôle d'accès basé sur les rôles (Lambda UpdateExpenseStatus)

Le JWT émis par Cognito contient le claim cognito:groups. La Lambda extrait ces groupes et vérifie le droit d'action avant tout accès à la base de données :

// Extraction des groupes depuis le JWT (injecté par API Gateway Cognito Authorizer)
var groups = request.RequestContext.Authorizer.Claims["cognito:groups"]
    .Split(',').Select(g => g.Trim()).ToList();
var isFinance = groups.Contains("finance");

// Actions réservées au service financier (approve, reject)
if (FinanceActions.Contains(action) && !isFinance)
    return Response(403, "Seul le service financier peut approuver ou rejeter.");

// Actions réservées au propriétaire (submit, resubmit)
if (OwnerActions.Contains(action) && item["UserId"].S != userId)
    return Response(403, "Vous ne pouvez modifier que vos propres dépenses.");

4.2 Machine à états server-side (Lambda UpdateExpenseStatus)

Toute tentative de transition invalide est rejetée HTTP 409, quel que soit ce que le client demande. Un verrou optimiste DynamoDB prévient les conflits de concurrence :

private static readonly Dictionary<(string, string), string> ValidTransitions = new()
{
    { ("Draft",       "submit"),   "Submitted"   },
    { ("Submitted",   "approve"),  "Approved"    },
    { ("Submitted",   "reject"),   "Rejected"    },
    { ("Rejected",    "resubmit"), "Resubmitted" },
    { ("Resubmitted", "submit"),   "Submitted"   },
};

var currentStatus = item["Status"].S;
if (!ValidTransitions.TryGetValue((currentStatus, action), out var newStatus))
    return Response(409, $"Transition invalide : '{action}' depuis '{currentStatus}'.");

// Verrou optimiste : échec si le statut a changé entre lecture et écriture
await _dynamoDb.UpdateItemAsync(new UpdateItemRequest {
    ConditionExpression = "#status = :currentStatus",
    // ... mise à jour du statut
});

4.3 URLs pré-signées S3 (Lambda GeneratePresignedUrl)

Le bucket S3 est entièrement privé. Lambda génère des URLs temporaires via le SDK AWS :

// Upload : URL PUT valable 15 minutes
var uploadUrl = await _s3.GetPreSignedURLAsync(new GetPreSignedUrlRequest {
    BucketName  = _bucketName,
    Key         = $"receipts/{userId}/{expenseId}/{fileName}",
    Verb        = HttpVerb.PUT,
    Expires     = DateTime.UtcNow.AddMinutes(15),
    ContentType = "image/jpeg"
});

// Download : URL GET valable 60 minutes
var downloadUrl = await _s3.GetPreSignedURLAsync(new GetPreSignedUrlRequest {
    BucketName = _bucketName,
    Key        = objectKey,
    Verb       = HttpVerb.GET,
    Expires    = DateTime.UtcNow.AddHours(1)
});

La clé S3 est enregistrée dans DynamoDB (ReceiptKey) dès la génération de l'URL d'upload, permettant à l'application de savoir si un justificatif est attaché.

5. Conclusion

Difficultés rencontrées et solutions adoptées

  • Compilation XAML sous Windows — Le compilateur XamlCompiler.exe échouait silencieusement. Problème combiné : mauvais namespace XML dans Platforms/Windows/App.xaml (2009/xaml au lieu de 2006/xaml) et propriété WindowsPackageType=None manquante. Résolution par comparaison avec un projet template dotnet new maui.
  • Erreur 401 sur l'API Gateway — L'application envoyait l'AccessToken Cognito. Or, le Cognito Authorizer valide l'IdToken qui contient les claims utilisateur (email, groupes). Correction en une ligne dans ExpenseService.cs.
  • File Finance incomplète — La Lambda ne retournait que les dépenses Submitted, excluant les Resubmitted. Correction dans FinanceViewModel par deux appels fusionnés.
  • ImplicitUsings manquant dans les projets Lambda — Types Task<> et Random non résolus. Ajout de <ImplicitUsings>enable</ImplicitUsings> dans chaque .csproj.

Bilan fonctionnel

L'application déployée sur AWS (eu-west-1) couvre l'intégralité du workflow demandé : création de dépenses (Draft/Submitted), validation ou rejet avec commentaire obligatoire, resoumission après rejet, upload de justificatifs via URL pré-signée. La machine à états est strictement enforced côté serveur. Le RBAC est appliqué dès l'entrée dans chaque Lambda, avant tout accès à DynamoDB.

Améliorations possibles

  • Notifications email — Intégrer Amazon SES pour notifier l'employé lors d'une approbation ou d'un rejet.
  • Seuil de double validation — Exiger deux approbations pour les montants supérieurs à 500€, règle configurable dans Lambda.
  • Dashboard analytique — Exploiter le GSI CategoryIndex et Amazon QuickSight pour des rapports de dépenses par catégorie et période.
  • Mobile iOS/Android — Le projet MAUI est structuré multi-plateforme ; seule la configuration de build manque.
